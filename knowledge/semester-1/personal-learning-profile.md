# Personal Learning Profile: Semester 1 to StudyMate AI

## Purpose

This file summarizes Martin's personal learning angle from the first semester and how it connects to StudyMate AI.

It should help StudyMate AI answer questions about learning progress, exam preparation, project reflection, and portfolio explanation in a personal but honest way.

## Learning Position

Martin is a Multimedia Design student who wants to connect design, UX, frontend code, documentation, and AI into projects that are realistic to build and easy to explain.

The preferred project level is stronger than a basic beginner project, but still simple enough to understand, maintain, and present clearly.

Important boundaries:

- Use HTML, CSS, and vanilla JavaScript by default.
- Avoid React, TypeScript, databases, login, and advanced architecture unless there is a clear reason.
- Prioritize working software over complicated structure.
- Keep explanations practical and exam-ready.

## Personal Focus

Martin's personal focus is to build portfolio projects that show both visual design and technical understanding.

StudyMate AI is used as a portfolio demo to show that Martin can:

- Design a useful AI product concept.
- Build a frontend with a real chat interface.
- Build a simple backend with Express.
- Protect an API key on the backend.
- Connect OpenAI safely through an API route.
- Structure assistant roles and prompts.
- Use a local Knowledge Base instead of relying only on a generic chatbot.

## Exam and Portfolio Explanation

A strong explanation should connect:

- The problem: generic AI tools can feel unfocused.
- The solution: six focused assistants for different study tasks.
- The UX choice: the user chooses an assistant instead of writing a long role prompt every time.
- The technical choice: the backend owns assistants, skills, Knowledge Base, prompt building, rate limiting, and OpenAI calls.
- The security choice: the OpenAI key stays in `backend/.env` and is never exposed in frontend JavaScript.
- The limitation: this is a portfolio demo, not a production SaaS product.

## Honest Weaknesses

Current weaknesses that should be explained honestly:

- The Knowledge Base is still early and needs more personal project summaries.
- Knowledge matching is keyword based, not semantic.
- Debug endpoints should be disabled before public launch.
- Rate limiting is simple and in memory.
- There is no login, database, or cloud chat history.
- Real users have not tested the product deeply yet.
